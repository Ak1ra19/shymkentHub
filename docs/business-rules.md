# Бизнес-правила проекта

## CRM-система бронирования коворкинга

Этот документ описывает бизнес-логику системы: правила, ограничения, проверки, статусы, права доступа и поведение модулей.

Документ является источником правды для реализации бизнес-логики. Если поведение системы отличается от этого файла, необходимо либо исправить реализацию, либо обновить документацию в рамках той же задачи.

---

## 1. Общие правила

1. Система используется для бронирования ресурсов коворкинга.
2. Основные ресурсы системы:
   - рабочие места;
   - конференц-зал;
   - мероприятия.
3. Все действия пользователя должны выполняться только после авторизации, кроме регистрации и страницы входа.
4. Доступ к административным функциям разрешен только пользователям с ролью администратора.
5. Пользователь не должен иметь доступ к данным других пользователей, если это не разрешено отдельным правилом.
6. Администратор может просматривать и управлять бронированиями всех пользователей.

---

## 2. Роли и доступы

### 2.1 User

Пользователь может:

- просматривать главную страницу;
- просматривать календарь мероприятий;
- бронировать рабочие места;
- отправлять заявки на бронирование конференц-зала;
- просматривать свои бронирования;
- просматривать статусы своих заявок;
- редактировать свой профиль;
- отменять свои бронирования, если отмена разрешена системой.

Пользователь не может:

- управлять чужими бронированиями;
- видеть персональные данные других пользователей;
- одобрять или отклонять заявки;
- выгружать административные отчеты;
- управлять мероприятиями;
- управлять пользователями.

### 2.2 Admin

Администратор может:

- просматривать все бронирования;
- управлять заявками на конференц-зал;
- одобрять заявки;
- отклонять заявки;
- просматривать календарь бронирований;
- фильтровать бронирования;
- экспортировать отчеты;
- создавать и редактировать мероприятия;
- управлять пользователями;
- блокировать пользователей;
- просматривать аналитику;
- управлять режимом работы общего зала.

Журнал действий ведется системой как внутренняя история событий и не обязан отображаться отдельным пунктом навигации.

---

## 3. Регистрация

1. При регистрации пользователь обязан заполнить:
   - ФИО;
   - ИИН;
   - номер телефона;
   - должность;
   - компанию.
2. ИИН является персональным идентификатором и должен храниться в защищенном виде.
3. На форме регистрации должен быть доступен PDF-файл `Инструкция для резидентов`.
4. Пользователь обязан подтвердить ознакомление с правилами через чекбокс `Я ознакомлен с правилами`.
5. Если чекбокс не подтвержден, регистрация должна быть заблокирована.
6. После успешной регистрации пользователь получает роль `User`, если иное не задано администратором.

---

## 4. Авторизация

1. Пользователь может войти в систему по номеру телефона или ИИН.
2. Система может поддерживать следующие способы входа:
   - пароль;
   - Google account;
   - iCloud.
3. Заблокированный пользователь не должен иметь возможность войти в систему.
4. После авторизации пользователь должен видеть только доступные ему разделы.

---

## 5. Рабочие места

### 5.1 Количество рабочих мест

1. По умолчанию в справочнике создается 60 рабочих мест.
2. Фактическое количество рабочих мест определяется активными записями справочника и не должно быть жестко зафиксировано в коде.
3. Администратор управляет рабочими местами через Filament: добавляет, массово добавляет последовательные номера, редактирует, включает, отключает и удаляет места без истории бронирований.
4. Каждое рабочее место должно иметь уникальный номер.
5. Пользователь должен выбирать конкретное активное рабочее место при бронировании.
6. Администратор может закрепить рабочее место за конкретным пользователем.
7. Закрепленное рабочее место доступно для бронирования только пользователю, за которым оно закреплено.
8. Незакрепленные рабочие места доступны всем пользователям.
9. Администратор управляет режимом работы общего зала через Filament: указывает дату начала действия, время начала и время окончания рабочего дня.
10. Если для даты нет отдельной настройки режима работы, используется рабочий день `09:00 - 18:00`.

### 5.2 Дата бронирования

1. Рабочее место можно бронировать только на текущий день.
2. Бронирование на будущую дату запрещено.
3. Бронирование на прошедшую дату запрещено.
4. Если дата бронирования не равна текущей дате, система должна отклонить бронирование.

### 5.3 Время бронирования

1. Пользователь должен указать время начала и время окончания бронирования.
2. Время окончания должно быть позже времени начала.
3. Бронирование не должно пересекаться с другим активным бронированием того же рабочего места.
4. При выборе рабочего места система должна показать занятые интервалы на текущий день.
5. При выборе рабочего места система должна автоматически подставить ближайший свободный часовой интервал.
6. Если пользователь меняет время начала, время окончания автоматически становится на 1 час позже.
7. Если время окончания не указано при сохранении, система использует интервал 1 час от времени начала.
8. Время бронирования должно находиться внутри действующего рабочего дня общего зала.
9. Пользователь может выбрать бронирование на весь рабочий день; система использует действующие начало и окончание рабочего дня.
10. Если рабочее место занято на весь рабочий день, карта мест должна показывать отдельное визуальное состояние.

### 5.4 Конфликты бронирования

1. Одно рабочее место не может быть забронировано двумя пользователями одновременно.
2. Система должна проверять пересечения по:
   - номеру рабочего места;
   - дате;
   - времени начала;
   - времени окончания;
   - активному статусу бронирования.
3. При конфликте система должна отказать в создании бронирования.
4. Если выбранное время пересекается с активным бронированием, система должна отказать в создании бронирования.

### 5.5 Статусы бронирования рабочего места

Для бронирования рабочего места используются статусы:

- `Активно` — бронирование успешно создано;
- `Отменено` — пользователь или администратор отменил бронирование;
- `Завершено` — время бронирования прошло.

---

## 6. Конференц-зал

### 6.1 Общие правила

1. Конференц-зал рассчитан на 12 человек.
2. Пользователь должен указать:
   - дату;
   - время начала;
   - время окончания;
   - цель бронирования.
3. После отправки заявка должна попасть в админ-панель на согласование.
4. Бронирование конференц-зала не подтверждается автоматически.

### 6.2 Ограничение по времени

1. Один пользователь может бронировать конференц-зал максимум на 2 часа в день.
2. Ограничение применяется к суммарному времени всех активных заявок пользователя за один день.
3. В расчет лимита входят заявки со статусами:
   - `На рассмотрении`;
   - `Одобрено`.
4. Заявки со статусами `Отклонено` и `Отменено` не должны учитываться в дневном лимите.
5. Если новая заявка превышает лимит 2 часа в день, система должна отклонить создание заявки.

### 6.3 Конфликты по времени

1. Система должна проверять конфликты по времени для конференц-зала.
2. Конфликт возникает, если новая заявка пересекается по времени с другой активной заявкой на тот же зал.
3. Активными считаются заявки со статусами:
   - `На рассмотрении`;
   - `Одобрено`.
4. Если найден конфликт, система должна запретить создание новой заявки или показать пользователю недоступность выбранного времени.
5. При создании заявки система должна показывать занятые интервалы конференц-зала на выбранную дату.
6. При выборе даты система должна автоматически подставить ближайший свободный часовой интервал.
7. Если пользователь меняет время начала, время окончания автоматически становится на 1 час позже.
8. Если время окончания не указано при сохранении, система использует интервал 1 час от времени начала.

### 6.4 Статусы заявки на конференц-зал

Для заявки на конференц-зал используются статусы:

- `На рассмотрении` — заявка создана и ожидает решения администратора;
- `Одобрено` — администратор подтвердил заявку;
- `Отклонено` — администратор отказал в заявке;
- `Отменено` — заявка отменена пользователем или администратором;
- `Завершено` — время одобренного бронирования прошло.

### 6.5 Решение администратора

1. Только администратор может одобрить или отклонить заявку на конференц-зал.
2. При одобрении система должна изменить статус заявки на `Одобрено`.
3. При отклонении система должна изменить статус заявки на `Отклонено`.
4. При отклонении желательно указать комментарий администратора.
5. Комментарий администратора должен быть виден пользователю в карточке заявки.

---

## 7. Мои бронирования

1. Пользователь должен видеть только свои бронирования в профиле пользователя.
2. В списке бронирований должны отображаться:
   - тип бронирования;
   - рабочее место или конференц-зал;
   - номер рабочего места, если применимо;
   - дата;
   - время начала;
   - время окончания;
   - статус.
3. Пользователь должен видеть актуальный статус заявки на конференц-зал.
4. Если заявка отклонена, пользователь должен видеть комментарий администратора, если он был указан.

---

## 8. Админ-панель

### 8.1 Dashboard

На dashboard должны отображаться:

- количество бронирований за текущий день;
- количество занятых рабочих мест;
- количество заявок на согласование;
- ближайшие мероприятия.
- оперативная сводка по сегодняшним броням и ближайшим заявкам, если она доступна в интерфейсе.

### 8.2 Управление рабочими местами

Администратор должен видеть по каждому бронированию:

- пользователя;
- компанию пользователя;
- номер рабочего места;
- дату;
- время начала;
- время окончания;
- статус.

### 8.3 Управление конференц-залом

Администратор должен видеть по каждой заявке:

- ФИО пользователя;
- компанию;
- дату;
- время начала;
- время окончания;
- цель бронирования;
- статус;
- комментарий администратора, если он есть.

---

## 9. Календарь бронирований

1. Администратор должен видеть все бронирования в календарном формате.
2. Календарь должен поддерживать фильтрацию:
   - по конкретной дате;
   - по периоду `от` и `до`;
   - по типу бронирования.
3. Типы бронирования:
   - рабочее место;
   - конференц-зал.
4. Пользовательский календарь должен показывать мероприятия и доступные пользователю данные.
5. Обычный пользователь не должен видеть чужие персональные бронирования.

---

## 10. Отчеты и экспорт Excel

1. Экспорт доступен только администратору.
2. Формат экспорта: `.xlsx`.
3. Администратор может сформировать:
   - отчет за конкретный день;
   - отчет за выбранный период.
4. В отчете должны быть поля:
   - ФИО пользователя;
   - компания;
   - тип бронирования;
   - рабочее место;
   - конференц-зал;
   - номер стола / рабочего места;
   - дата;
   - время начала;
   - время окончания;
   - количество часов;
   - статус бронирования.
5. Количество часов рассчитывается как разница между временем окончания и временем начала.
6. Если бронирование отменено, оно может отображаться в отчете со статусом `Отменено`, если администратор не применил фильтр, исключающий отмененные записи.

---

## 11. Мероприятия

1. Только администратор может создавать мероприятия.
2. Для мероприятия обязательны:
   - название;
   - описание;
   - дата;
   - время.
3. Фото или баннер мероприятия является желательным полем.
4. После создания мероприятие должно автоматически отображаться в пользовательском календаре.
5. Пользователь не может создавать, редактировать или удалять мероприятия, если ему не выдана административная роль.

---

## 12. Уведомления

Система должна отправлять уведомления при событиях:

- успешное бронирование рабочего места;
- создание заявки на конференц-зал;
- одобрение заявки;
- отклонение заявки;
- отмена бронирования;
- изменение статуса бронирования.

Каналы уведомлений:

- внутри системы;
- Google account;
- iCloud;
- Email, если канал включен.

Если внешний канал уведомлений недоступен, системное уведомление внутри приложения должно оставаться основным обязательным каналом.

---

## 13. Отмена бронирования

1. Пользователь может отменить свое бронирование, если этот функционал включен.
2. Администратор может отменить любое бронирование.
3. После отмены статус должен измениться на `Отменено`.
4. Отмененное бронирование не должно блокировать рабочее место или конференц-зал.
5. После отмены пользователь должен получить уведомление.

---

## 14. Блокировка пользователя

1. Администратор может заблокировать пользователя.
2. Заблокированный пользователь не должен иметь возможность авторизоваться.
3. Заблокированный пользователь не должен иметь возможность создавать новые бронирования.
4. История бронирований заблокированного пользователя должна сохраняться.

---

## 15. Безопасность персональных данных

1. ИИН должен храниться в защищенном виде.
2. Обычные пользователи не должны иметь доступ к ИИН других пользователей.
3. Административные разделы должны быть защищены проверкой роли.
4. Система не должна раскрывать персональные данные в публичных страницах.
5. Доступ к отчетам должен быть разрешен только администраторам.

---

## 16. Журнал действий

Если журнал действий реализован, в нем желательно фиксировать:

- создание бронирования;
- отмену бронирования;
- создание заявки на конференц-зал;
- одобрение заявки;
- отклонение заявки;
- создание мероприятия;
- изменение данных пользователя;
- блокировку пользователя.

---

## 17. Правила изменения бизнес-логики

Бизнес-логикой считаются изменения, которые затрагивают:

- расчеты;
- правила валидации;
- статусы;
- связи между сущностями;
- роли и права доступа;
- бронирования;
- отмены;
- уведомления;
- отчеты;
- мероприятия;
- календарь;
- лимиты и ограничения.

При изменении бизнес-логики необходимо обновить:

- `docs/business-rules.md`;
- соответствующий файл модуля в `docs/modules/*.md`;
- `docs/changelog.md`.

Задача не считается завершенной, пока связанная документация не обновлена.
