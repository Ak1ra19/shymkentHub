# Техническое задание

## CRM-система бронирования коворкинга

Система представляет собой CRM / Web-приложение для бронирования рабочих мест, конференц-зала, управления мероприятиями, пользователями, заявками и отчетами коворкинга.

Документация является основным источником правды для бизнес-логики проекта. Перед реализацией любой новой функции необходимо сверяться с этим файлом, `docs/business-rules.md` и соответствующими файлами модулей в `docs/modules/*.md`.

---

## 1. Цель системы

Цель системы — автоматизировать процесс бронирования ресурсов коворкинга:

- рабочих мест;
- конференц-зала;
- мероприятий;
- заявок пользователей;
- административного контроля;
- отчетности по бронированиям и посещаемости.

---

## 2. Роли пользователей

### 2.1 Пользователь

Пользователь — резидент или посетитель коворкинга, который может пользоваться личным кабинетом и бронировать доступные ресурсы.

Пользователь должен иметь возможность:

- зарегистрироваться в системе;
- авторизоваться;
- просматривать календарь мероприятий;
- бронировать рабочие места;
- отправлять заявки на бронирование конференц-зала;
- просматривать свои бронирования;
- видеть статусы заявок;
- получать уведомления;
- редактировать профиль;
- отменять свои бронирования, если это разрешено правилами системы.

### 2.2 Администратор

Администратор — сотрудник коворкинга, который управляет пользователями, бронированиями, календарем, мероприятиями и отчетами.

Администратор должен иметь возможность:

- просматривать dashboard;
- управлять бронированиями рабочих мест;
- рассматривать заявки на конференц-зал;
- одобрять или отклонять заявки;
- просматривать календарь всех бронирований;
- фильтровать бронирования;
- выгружать отчеты в Excel;
- управлять мероприятиями;
- управлять пользователями;
- блокировать пользователей;
- просматривать аналитику и журнал действий.

---

## 3. Регистрация и авторизация

### 3.1 Регистрация пользователя

При регистрации пользователь должен заполнить следующие поля:

- ФИО;
- ИИН;
- номер телефона;
- должность;
- компания.

Дополнительно на форме регистрации должен быть доступен PDF-файл:

- `Инструкция для резидентов`.

Пользователь должен:

- открыть или прочитать PDF-файл;
- поставить чекбокс `Я ознакомлен с правилами`.

Без подтверждения чекбокса регистрация невозможна.

### 3.2 Авторизация

Пользователь должен иметь возможность войти в систему:

- по номеру телефона;
- или по ИИН.

Способы авторизации:

- пароль;
- Google account;
- iCloud.

---

## 4. Пользовательская панель

### 4.1 Главная страница

На главной странице пользователя должны отображаться:

- календарь мероприятий;
- ближайшие мероприятия;
- новости или анонсы, если этот функционал включен.

## 5. Бронирование рабочих мест

### 5.1 Описание

В коворкинге по умолчанию доступно 60 рабочих мест, созданных в справочнике рабочих мест. Фактическое количество мест не должно быть жестко зафиксировано в коде: администратор управляет местами через Filament, добавляя, отключая или удаляя записи справочника.

Пользователь должен иметь возможность:

- выбрать текущий день;
- выбрать время бронирования;
- выбрать конкретное активное рабочее место из справочника;
- видеть занятые интервалы выбранного рабочего места;
- создать бронирование.

### 5.2 Ограничения

Бронирование рабочих мест доступно только на сегодняшний день.

Пользователь не может:

- бронировать рабочее место на будущую дату;
- бронировать рабочее место на прошедшую дату;
- забронировать уже занятое место на пересекающееся время.

Система должна:

- автоматически показывать свободные и занятые места;
- при выборе места автоматически подставлять ближайший свободный часовой интервал;
- показывать занятые интервалы выбранного места в форме бронирования;
- автоматически ставить время окончания на 1 час позже времени начала, если пользователь изменил начало или не указал окончание;
- не допускать двойного бронирования одного места на одно и то же время.

Администратор может закрепить рабочее место за конкретным пользователем. Закрепленное место доступно для бронирования только этому пользователю, незакрепленные места доступны всем пользователям.

### 5.3 Мои бронирования

После успешного бронирования данные должны сохраняться в разделе `Мои бронирования`.

В разделе должны отображаться:

- номер рабочего места;
- дата;
- время начала;
- время окончания;
- статус бронирования.

---

## 6. Бронирование конференц-зала

### 6.1 Описание

Конференц-зал рассчитан на 12 человек.

Пользователь должен иметь возможность:

- выбрать дату;
- выбрать время начала;
- выбрать время окончания;
- видеть занятые интервалы зала на выбранную дату;
- указать цель бронирования.

Пример цели бронирования:

> Мероприятие по Frontend разработке

### 6.2 Ограничения

Для конференц-зала действуют ограничения:

- максимум 2 часа бронирования в день на одного пользователя;
- система должна проверять конфликты по времени;
- при выборе даты система должна предлагать ближайший свободный часовой интервал;
- при изменении времени начала система должна автоматически ставить окончание на 1 час позже;
- заявка после отправки уходит в админ-панель на согласование.

### 6.3 Статус заявки

В панели пользователя должен отображаться статус заявки на конференц-зал:

- `На рассмотрении`;
- `Одобрено`;
- `Отклонено`.

При отклонении желательно показывать комментарий администратора.

---

## 7. Админ-панель

### 7.1 Dashboard

Главная страница администратора должна показывать:

- количество бронирований за день;
- количество занятых рабочих мест;
- количество заявок на согласование;
- ближайшие мероприятия.

### 7.2 Управление бронированием рабочих мест

Администратор должен видеть:

- кто забронировал рабочее место;
- номер рабочего места;
- дату бронирования;
- время начала;
- время окончания;
- статус бронирования.

Отображение должно быть доступно в двух форматах:

- календарный формат;
- список бронирований.

### 7.3 Управление конференц-залом

Администратор должен иметь возможность:

- видеть новые заявки;
- просматривать данные заявки;
- одобрять заявку;
- отклонять заявку;
- указывать комментарий при отказе.

В заявке должны отображаться:

- ФИО пользователя;
- компания;
- дата;
- время начала;
- время окончания;
- цель бронирования;
- текущий статус.

### 7.4 Календарь бронирований

В админ-панели должен быть календарь бронирований.

Календарь должен поддерживать:

- просмотр всех бронирований;
- фильтр по конкретной дате;
- фильтр по периоду `от` и `до`;
- сортировку или фильтр по типу бронирования:
  - рабочие места;
  - конференц-зал.

---

## 8. Экспорт данных и отчеты

Администратор должен иметь возможность выгружать отчеты по бронированиям в формате Excel `.xlsx`.

### 8.1 Отчет за конкретный день

Администратор выбирает определенный день и получает список всех бронирований за этот день.

### 8.2 Отчет за период

Администратор выбирает диапазон дат:

- дата `от`;
- дата `до`.

После выбора периода система формирует полный отчет по всем бронированиям за выбранный диапазон.

### 8.3 Поля отчета

В отчете должны отображаться:

- ФИО пользователя;
- компания;
- тип бронирования;
- рабочее место;
- конференц-зал;
- номер стола / рабочего места;
- дата;
- время начала;
- время окончания;
- количество часов;
- статус бронирования.

---

## 9. Управление мероприятиями

Администратор должен иметь возможность добавлять мероприятие.

Поля мероприятия:

- название;
- описание;
- дата;
- время;
- фото или баннер мероприятия.

После добавления мероприятие автоматически отображается в пользовательском календаре.

---

## 10. Уведомления

Система должна отправлять уведомления пользователю при следующих событиях:

- успешное бронирование;
- изменение статуса заявки;
- отмена бронирования.

Каналы уведомлений:

- внутри системы;
- Google account;
- iCloud;
- Email, если этот канал включен.

## 12. Технические требования

### 12.1 Формат системы

Система должна быть реализована как Web application.

### 12.2 Адаптивность

Интерфейс должен корректно работать на:

- desktop;
- mobile version.

### 12.3 Безопасность

Система должна учитывать требования безопасности:

- защита персональных данных;
- хранение ИИН в защищенном виде;
- разграничение доступа по ролям;
- авторизация пользователей;
- защита административных разделов от доступа обычных пользователей.
